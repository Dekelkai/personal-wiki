# AGENTS.md

本文件约束所有参与维护本知识库的 Agent。正式知识源是 Markdown、YAML Frontmatter 与 Git。

## 项目上下文

1. 开始较大任务前阅读 `docs/项目总览与路线图.md`。
2. 创建或实质性修改知识页面前阅读 `docs/内容与排版规范.md`。
3. 处理论文、PDF、学习笔记、实验记录或其他外部材料时，使用仓库 Skill `$personal-wiki-curator`；默认先进入只读分析模式，不直接写入正式 Wiki。
4. Quartz 是公开展示层，不是知识库本体。不得为了页面效果破坏知识对象、来源追踪或公开与私有边界。

## 修改前

1. 先搜索已有页面、标题、别名和近义概念。
2. 优先补充已有页面，不重复创建近义页面。
3. 修改现有文件前，在项目根目录 `.backup/<时间戳>/` 按原相对路径备份。
4. 不确定的信息必须标注“待验证”，不得把推测写成事实。
5. 引用外部资料时记录来源。
6. 先判断输入是原始材料、来源证据还是正式知识，不得把原始笔记或报告原样复制成 Wiki 页面。

## 禁止事项

1. 禁止直接删除正式笔记。
2. 禁止批量移动目录。
3. 禁止未经确认修改 `publish` 字段。
4. 禁止将 `_inbox` 内容自动公开。
5. 禁止修改、移动、删除或批量复制 `F:\OneDrive\obsidian\PaperNotes`、`F:\OneDrive\obsidian\Notes` 和 `F:\Blog` 中的文件。
6. 禁止把 API Key、Token、密码、Cookie、私钥、私人服务器地址等敏感信息写入知识库。
7. 禁止创建模糊名称或为每个失效 Wiki Link 自动创建空页面。
8. 禁止因为页面设置了 `publish: false` 就将未公开论文、组内资料或敏感实验数据写入公开仓库。

## 外部材料读取

1. 外部材料由用户保留在原有位置，Agent 只读取用户明确给出的文件或目录，不进行无边界扫描。
2. Windows 路径在 WSL 中按挂载路径读取，例如 `F:\OneDrive\obsidian\Notes\...` 对应 `/mnt/f/OneDrive/obsidian/Notes/...`。
3. `PaperNotes` 用于私人论文阅读笔记，`Notes` 用于报告、专题笔记和其他原始资料，Zotero 用于论文元数据、PDF、CiteKey、DOI 与批注，`Blog` 用于叙事型输出。
4. 外部材料只作为输入和证据；需要进入 Wiki 的是经过识别、去重、综合和来源标注的知识对象。

## 内容规范

1. 文件命名、Frontmatter、Wiki Link 和 Emoji 遵守项目 README 与 `schemas/frontmatter.schema.json`。
2. Emoji 不进入文件名、路径、Frontmatter 字段名或枚举值。
3. 正式页面必须有实质内容，领域页和索引页负责组织，具体页面负责解释。新建的读者可见页面名称默认不使用“地图”。
4. 对正文做实质性重写时保留用户原始观点，不得将推测改写为事实。
5. Markdown 文件统一使用 UTF-8。
6. 明确区分外部事实、来源作者结论、用户实验结果、用户判断和 Agent 推断。
7. 创建页面时以当前 Schema 和模板为准；目标知识模型与当前 Schema 不一致时先报告，不自行扩展枚举。

## Skill 维护

1. `$personal-wiki-curator` 是可迭代工作流，不是固定模板。每次材料提炼任务结束时都要输出 Skill 缺口复盘。
2. 只有可复用的问题才进入 Skill：流程规则写入 `SKILL.md`，详细判断标准写入 `references/`，确定性且重复的机械检查才考虑脚本。
3. 安全、事实边界或数据损坏风险应立即修正规则；单次措辞和个人风格问题先记录，重复出现后再固化。
4. 不得静默自我修改 Skill。当前任务未明确包含 Skill 优化时，只提出建议并等待用户确认。
5. 修改 Skill 后运行其结构验证，并说明修改原因和适用范围。

## 完成与交付

1. 修改完成后运行 `python scripts/check_kb.py`。
2. 修改知识正文、模板、Schema 或 Quartz 配置后运行 Quartz 构建，并检查实际生成结果。
3. 根据检查结果修复本轮造成的错误，再次运行检查。
4. 最终输出修改摘要、创建文件列表、修改文件列表和仍未解决的问题。
5. 未经明确要求不得执行 `git push`。
