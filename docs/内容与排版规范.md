# 个人 Wiki 内容与排版规范

> 本文档定义正式知识页面的内容契约和 Quartz 展示约束。它面向维护者与 Agent，不进入 `content/`，也不由 Quartz 发布。

最后更新：2026-07-16

## 1. 规范读取顺序

执行材料提炼或页面修改任务时，按以下顺序确认规则：

1. `AGENTS.md`：权限、安全、备份和交付规则；
2. `docs/项目总览与路线图.md`：项目目标、分类和阶段路线；
3. 本文档：内容结构、来源标注和 Quartz 排版；
4. `schemas/frontmatter.schema.json`：当前可执行的 Frontmatter 约束；
5. `content/_templates/` 中对应模板；
6. 目标页面、同领域页面和相关索引页；
7. `quartz.config.yaml`：实际公开展示配置。

文档中的目标模型不能覆盖当前 Schema。若二者不一致，先报告并提出迁移方案，不直接创建使用未支持枚举的页面。

## 2. 材料放置与读取

原始材料不需要复制到 personal-wiki。用户提供明确路径后，Agent 从原位置只读分析：

| 来源 | Windows 路径或系统 | WSL 路径 | 角色 |
| --- | --- | --- | --- |
| 专题笔记与报告 | `F:\OneDrive\obsidian\Notes` | `/mnt/f/OneDrive/obsidian/Notes` | PDF、专题报告、学习材料和临时综合 |
| 论文阅读笔记 | `F:\OneDrive\obsidian\PaperNotes` | `/mnt/f/OneDrive/obsidian/PaperNotes` | 私人阅读过程和未核验理解 |
| 文献管理 | Zotero | 由用户指定 | PDF、元数据、CiteKey、DOI 和批注 |
| 博客 | `F:\Blog` | `/mnt/f/Blog` | 已发布或准备发布的叙事型文章 |

读取规则：

- 只读取用户明确指定的文件；需要读取相邻文件时先说明原因；
- 不修改、移动、重命名或批量复制外部材料；
- 不把完整 PDF、批注、原始笔记或大段摘录提交进仓库；
- 从外部材料提取的事实必须保留文件名、页码、章节、表格或其他定位信息；
- 来源不足、页面不可辨认或不同来源冲突时标注“待验证”。

## 3. 领域与知识对象

目录表达主要领域，Frontmatter `type` 表达页面对象。读者可见的主要领域是：

- 当前研究；
- 计算机科学与工程；
- 人工智能；
- 项目与实验；
- 写作与输出；
- 参考资料。

正式页面不按“每份材料一个页面”机械生成。先判断内容属于 `concept`、`method`、`dataset`、`paper`、`experiment` 等对象，再决定更新已有页面还是创建新页面。

当前 Schema、模板和检查脚本已经支持 `topic`、`metric` 和 `survey`。创建这些页面时仍需满足对象边界稳定、内容实质充分和来源可追踪等条件，不得因为类型可用就批量创建空页面。

### 更新已有页面还是创建新页面

优先更新已有页面，当且仅当候选内容与现有页面具有相同的核心定义、对象边界和主要用途。

可以建议新建页面的条件：

1. 已搜索标题、别名和近义概念；
2. 对象边界稳定，可用一两句话明确说明页面职责；
3. 已有足够实质内容，不是只有标题和待办项；
4. 与已有页面不是同义重复，也不是仅因来源不同而拆分；
5. 存放目录、`type`、`domain` 和来源关系均可解释。

## 4. Frontmatter 契约

正式页面至少包含当前 Schema 要求的字段：

```yaml
---
title: 页面标题
type: concept
domain: research
status: draft
publish: false
created: 2026-07-16
updated: 2026-07-16
---
```

要求：

- `title` 不带 Emoji；
- Emoji 不进入文件名、路径、字段名或枚举值；
- 新页面默认 `status: draft`、`publish: false`；
- 未经人工确认不修改 `publish`；
- 日期使用 `YYYY-MM-DD`；
- 论文、数据集和外部资源使用已有字段保存 DOI、URL、作者、年份、许可和版本；
- 引用结构尚未固化时，不擅自发明互不兼容的 Frontmatter 字段。

## 5. 正文与 Quartz 排版

### 主标题

- 正文恰好保留一个 H1；
- H1 可以使用一个与内容匹配的 Emoji，例如 `# 🗃️ 数据集名称`；
- Frontmatter `title` 保持纯文本；
- `article-title` 插件保持关闭，避免 Frontmatter 标题与正文 H1 重复；
- 浏览器标题、搜索、站点地图和社交预览继续使用 Frontmatter 标题。

### 标题层级

- H2 表达页面的主要问题或稳定结构，H3 用于 H2 下的细分；
- 不跳级，不用加粗文本模拟标题；
- Emoji 可适度用于 H1 和少数主要 H2，不要求每个标题都有 Emoji；
- 新建的读者可见页面名称和正文标题默认不使用“地图”，使用“领域”“概览”“主题”或“索引”。旧文件路径的迁移需要单独确认。

### 段落、列表与表格

- 先用短段落解释结论，再用列表表达并列信息；
- 表格适合比较数据集、方法、配置和指标，但每列含义、单位和缺失值必须明确；
- 不把未经核验的表格数字改写为确定结论；
- 命令和代码使用带语言标识的 fenced code block，并说明环境、版本和适用范围；
- 数学符号使用 Quartz 已支持的 LaTeX 语法。

### 图片与附件

- 图片应服务于解释，不为装饰而复制原始材料；
- 优先保存用户自有或许可允许的图；外部图需要记录来源与许可；
- 图片引用使用相对路径，并遵守 Obsidian 与 Quartz 均可解析的写法；
- 从 PDF 截取的图表默认只用于分析，不自动提交到公开仓库。

## 6. 来源、证据与表述边界

页面中的陈述至少区分以下五类：

| 类别 | 推荐表述 |
| --- | --- |
| 外部事实 | 说明来源，并尽量定位到文档、页码或官方页面 |
| 来源作者结论 | 使用“作者报告”“论文认为”等归属表达 |
| 用户实验结果 | 连接具体实验页面，记录配置、版本和结果 |
| 用户判断 | 明确使用“我的判断”“当前选择”等第一人称边界 |
| Agent 推断 | 标注“推断”或“待验证”，不得伪装成来源事实 |

论文页面负责保存单一来源的元数据、贡献、报告结果和局限；方法、数据集、指标和主题页面负责综合多份来源。用户实验结果不得写成论文原始结果，论文报告结果也不得写成用户已复现结果。

## 7. Wiki Link 与公开边界

- 先确认目标页面真实存在，再创建 Wiki Link；
- 不为失效链接批量创建空页面；
- 公开页面原则上只链接 `publish: true` 页面；
- 尚未发布或尚未创建的对象在公开页面中使用普通文本；
- `publish: false` 只阻止 Quartz 输出，不提供仓库级保密；
- `_inbox`、`_templates`、`_archive` 保持不公开。

## 8. 常见对象的正文重点

| 对象 | 正文重点 |
| --- | --- |
| `concept` | 定义、边界、原理、示例、易混概念 |
| `topic` | 研究问题、任务边界、输入输出、假设、方法路线、证据与开放问题 |
| `method` | 目标、输入输出、假设、步骤、实现、评测、局限 |
| `dataset` | 来源许可、版本、规模、模态、坐标、划分、标注、任务、限制 |
| `metric` | 定义、公式、单位、方向、聚合、适用任务和可比条件 |
| `paper` | 元数据、研究问题、方法、数据、指标、报告结果、贡献、局限、来源 |
| `survey` | 范围、纳入标准、检索方式、分类维度、证据综合、空缺与局限 |
| `experiment` | 假设、环境、代码与数据版本、配置、过程、结果、误差和结论 |
| `guide` | 前置条件、步骤、验证、回滚、适用范围 |
| `troubleshooting` | 现象、环境、原因、排查、修复、验证和适用范围 |
| `project` | 目标、范围、设计、里程碑、实验、决策、资源和状态 |

具体章节可按对象调整，不为匹配模板而保留空章节。

数据页面中的许可证必须指向数据本身。代码仓库标注的 MIT、Apache 等许可证不能自动视为数据许可证；如果官方页面没有明确数据许可，写为“待验证”，并分别记录代码许可和数据访问方式。

## 9. Quartz 不变量与验证

除非构建失败且有明确证据，不修改以下配置：

- `locale: zh-CN`；
- `baseUrl: tangxk.cn/personal-wiki`；
- `explicit-publish` 启用；
- `article-title` 关闭；
- `_inbox`、`_templates`、`_archive` 位于 `ignorePatterns`。

修改正式知识内容后至少运行：

```bash
python3 scripts/check_kb.py
node ./quartz/bootstrap-cli.mjs build
test -f public/index.html
```

涉及公开页面标题或布局时，还要检查生成 HTML 中实际可见的 H1，而不是只检查 Markdown 或配置文件。

## 10. 与 Skill 的关系

`$personal-wiki-curator` 负责把上述规范转化为可重复执行的材料分析与知识提炼流程。每次真实任务结束后都要进行 Skill 缺口复盘：

1. 记录遇到的问题；
2. 判断它是一次性问题还是可复用规则；
3. 决定应更新 Skill、参考规范、项目规则还是检查脚本；
4. 说明修改原因；
5. 验证修改没有削弱安全、来源和发布边界。

安全与事实边界问题可以在已授权的 Skill 优化任务中立即修正；单次审美或措辞偏好不立即固化。
