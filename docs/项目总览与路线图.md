# 个人 Wiki 项目总览与路线图

> 本文档是 personal-wiki 的长期项目上下文，用于记录项目目标、知识分类、对象边界、维护原则和阶段路线。后续开始较大任务前，应先阅读本文档、`AGENTS.md` 和 `README.md`。

最后更新：2026-07-16

## 1. 项目定位

personal-wiki 是一个面向长期维护的个人知识系统，用于组织、关联、检索和复用科研知识、计算机知识、工程经验、AI 实践、项目记录和写作成果。

正式知识源是：

- Markdown；
- YAML Frontmatter；
- Git 历史。

Obsidian 是主要编辑工具，Quartz 是公开展示层，GitHub Pages 是当前部署渠道。Quartz 不是知识库本体，不应反向决定知识对象如何划分。

本项目不是：

- PaperNotes 的公开镜像；
- Zotero 元数据或 PDF 的复制仓库；
- AI 对话、命令输出和实验草稿的集中归档；
- Hexo 博客的双向同步副本；
- 为 MCP、向量数据库或 RAG 单独建设的数据集。

## 2. 内容来源与职责边界

### 原始资料

原始资料包括 PaperNotes、Zotero PDF 与批注、临时想法、AI 对话、实验草稿、故障过程、博客原稿和项目工作日志。

这些材料只作为输入和证据，不直接原样复制为正式 Wiki 页面。

### 外部来源

- `F:\OneDrive\obsidian\PaperNotes`：私人论文阅读笔记，只允许选择性读取和提炼，禁止 Agent 修改或批量复制。
- `F:\OneDrive\obsidian\Notes`：专题报告、课程或技术笔记等原始材料。用户提供明确文件路径后只读分析，禁止 Agent 修改或无边界扫描。
- Zotero：负责论文元数据、PDF、CiteKey、DOI、批注和文献来源。
- `F:\Blog`：Hexo 博客，负责已发布或准备发布的叙事型文章，禁止 Agent 修改。

### 规范知识库

`F:\KnowledgeBase\personal-wiki` 是整理后的正式知识库。Agent 可以在治理规则约束下修改该仓库，但不得因为页面设置了 `publish: false` 就把敏感原始材料写入仓库。

## 3. 知识分类原则

分类需要同时区分三个维度，不能只依赖一棵目录树。

### 3.1 知识领域：页面讨论什么

读者可见的主要领域采用以下结构：

```text
首页
├── 当前研究
├── 计算机科学与工程
├── 人工智能
├── 项目与实验
├── 写作与输出
└── 参考资料
```

正式页面使用“领域”“概览”“主题”“索引”等名称，不再使用“地图”作为读者可见的页面名称。

### 3.2 知识对象：页面本身是什么

核心对象类型包括：

| 类型 | 职责 |
| --- | --- |
| `index` | 组织领域入口和页面关系，不承担具体知识解释 |
| `concept` | 解释稳定概念、术语和基本原理 |
| `topic` | 定义研究主题、研究问题、任务边界和开放问题 |
| `method` | 说明方法目标、输入输出、假设、步骤、实现和局限 |
| `dataset` | 记录数据来源、许可、版本、划分、标注和适用任务 |
| `metric` | 说明评价指标的定义、计算、方向、聚合和比较限制 |
| `paper` | 保存论文元数据、作者贡献、报告结果、证据和局限 |
| `survey` | 按明确范围、纳入标准和证据组织多份来源 |
| `experiment` | 记录用户实际执行的实验、配置、结果和误差分析 |
| `project` | 组织项目目标、范围、里程碑、资源和当前状态 |
| `decision` | 记录实际决策的背景、候选方案、权衡和后续影响 |
| `guide` | 保存经过验证、可复用的操作流程 |
| `troubleshooting` | 保存问题现象、原因、排查过程、修复和验证方法 |
| `reference` | 保存命令、清单、术语和资源等快速参考内容 |
| `article` | 组织面向外部读者的完整叙事型文章或博客索引 |

目录表达主要领域或工作归属，Frontmatter `type` 表达对象性质。页面可以通过 Wiki Link 与多个领域建立关系，但只保留一个主要存放位置。

### 3.3 工作视角：知识如何被使用

- 当前研究：展示用户实际正在研究的问题；
- 项目与实验：记录知识在真实任务中的应用与验证；
- 写作与输出：将研究、工程和项目内容组合成论文、报告和博客；
- 参考资料：提供跨领域的命令、术语、清单和资源。

## 4. 当前研究范围

当前真正需要建设的核心研究方向是：

```text
计算机视觉
└── 视觉定位
    └── 无人机绝对视觉定位
```

无人机绝对视觉定位可以关联：

- GNSS 拒止环境下的任务定义；
- 航拍图像与正射影像匹配；
- 跨视角图像检索；
- 全局表征与局部特征匹配；
- 图像配准与几何一致性验证；
- PnP 和 6-DoF 位姿估计；
- 数据集、基准和评价指标；
- 用户实验、错误分析和可复现性。

目标分割、自然语言处理、大模型、火星视觉定位、无人机跟踪和视觉语言模型等方向，只有在产生真实内容或与当前研究形成稳定关系后才创建正式页面。尚未成熟的方向保留在私有建设计划中，不提前建立空页面。

### 算法与数据结构的对象划分

| 内容 | 推荐类型 | 示例 |
| --- | --- | --- |
| 数据结构定义 | `concept` | 数组、链表、哈希表、树、图、堆 |
| 算法范式 | `concept` 或 `method` | 分治、贪心、动态规划、回溯 |
| 具体算法 | `method` | 二分查找、Dijkstra、KMP、归并排序 |
| 复杂度知识 | `concept` | 时间复杂度、空间复杂度、均摊分析 |
| 语言实现经验 | `guide` | Go 中实现堆、Python 图遍历注意事项 |
| 性能比较 | `experiment` | 不同算法或实现的基准测试 |

单道算法题和临时解题过程属于原始学习材料。只有其中可复用的概念、方法和实现经验需要进入正式 Wiki。

## 7. 原始资料到正式知识的流程

处理任意论文、笔记、实验或项目材料时，遵循以下流程：

1. 识别来源性质和可信范围；
2. 搜索现有标题、别名和近义概念；
3. 识别其中的 topic、concept、method、dataset、metric、paper、experiment 等对象；
4. 优先更新已有页面；
5. 只有对象边界稳定且具有实质内容时才创建页面；
6. 区分外部事实、论文作者结论、用户实验结果、用户判断和 Agent 推断；
7. 为外部事实和论文结论保留来源；
8. 将实验和决策连接到具体项目；
9. 将成熟对象连接到相关领域页；
10. 默认保持 `publish: false`，公开状态由人工确认。

正式知识的目标不是重现原始笔记，而是形成可由多份来源持续补充的知识对象。

## 8. 公开与私有边界

- `publish: false` 只控制 Quartz 是否生成页面，不是敏感信息保护机制；
- 仓库中不得保存 API Key、Token、密码、Cookie、私钥、私人服务器地址、未公开论文、组内资料或敏感实验数据；
- `_inbox`、`_templates` 和 `_archive` 不进入公开网站；
- 公开页面中的 Wiki Link 原则上只能指向同样公开的页面；
- 未公开目标在正式公开页面中先使用普通文本，成熟并发布后再转换为 Wiki Link；
- Agent 不得自动修改 `publish`。

## 10. 阶段路线

### 阶段 0：规范对齐

目标：让项目说明、分类、Schema、模板、检查脚本和公开页面遵守同一套规则。

主要任务：

1. 将读者可见的“知识地图”名称调整为领域名称；
2. 将“工程技术”的范围明确为“计算机科学与工程”；
3. 在 AI 领域中区分学科知识和工具实践；
4. 增加 `topic`、`metric`、`survey` 的类型定义；
5. 增加研究指标的目录规划；
6. 修复公开页面指向 `publish: false` 页面的问题；
7. 更新 README、AGENTS 和脚本说明中的过时信息；
8. 让检查脚本验证公开链接边界和 Schema 一致性；
9. 核验 Obsidian 的新文件位置和模板目录设置；
10. 建立内容与 Quartz 排版规范，以及可迭代的材料提炼 Skill。

完成标准：项目规则不存在明显冲突，公开页面没有指向未发布目标的链接，后续 Agent 能根据文档正确判断知识对象和存放位置。

### 阶段 1：小规模研究知识提炼试点

建议输入：

- 3 篇围绕无人机跨视角绝对视觉定位的论文；
- 1 份数据集官方说明；
- 1 份用户已有实验或复现记录。

预期输出：

- 1 个研究主题；
- 3 个论文证据页；
- 2 至 3 个方法页；
- 1 个数据集页；
- 1 至 2 个指标页；
- 1 个实验页。

完成标准：每个综合结论可回到来源，论文报告与用户实验清楚分离，第二批材料能够复用同一处理流程。

### 阶段 2：固化知识模型与 Zotero 最小接入

根据试点结果调整 Schema、模板、命名和来源字段，建立 CiteKey、DOI 和 Zotero 元数据的最小连接。此阶段只解决来源追踪，不自动复制 PDF、批注或完整笔记。

### 阶段 3：小批量扩展真实内容

每批处理少量相关材料，逐渐形成方法、数据集、指标、综述和实验网络。领域页只链接已经存在且具有实质内容的页面。

### 阶段 4：项目、实验与写作闭环

将研究主题连接到实际项目、实验、技术决策、论文写作和博客输出，验证知识库能否支持真实研究工作。

### 阶段 5：选择性自动化

只有当手工流程稳定后，才实现元数据同步、候选对象提取、选择性导入和自动检查。所有内容拆分、事实判断和公开状态仍需人工确认。

### 阶段 6：检索、MCP 与 RAG

只有当知识库已经包含足够的真实、结构化、可追踪内容，并且存在明确的检索与问答需求后，才评估 MCP、全文检索增强、向量检索和 RAG。

## 12. 材料提炼与 Skill

仓库级 Skill 位于 `.agents/skills/personal-wiki-curator/`。它用于处理用户明确指定的论文、PDF、笔记、实验记录和技术材料。

Skill 的默认行为是：

1. 阅读项目规则与当前 Schema；
2. 只读检查材料并保留来源定位；
3. 搜索已有知识对象；
4. 提出更新或新建建议；
5. 等待人工确认后再修改正式内容；
6. 修改时备份并运行检查与构建；
7. 任务末尾复盘 Skill 是否遗漏了可复用规则。

Skill 不是一成不变的。安全、事实边界和重复出现的流程问题应逐步固化；一次性措辞和审美偏好不应立即写成永久规则。Skill 修改必须可审查，并说明原因与适用范围。

## 14. 长期维护规则

- 先搜索，再新建；
- 先识别对象，再选择目录；
- 一个页面只有一个主要职责；
- 不创建只有标题和待补充项的正式页面；
- 不把论文页面当作全部知识的容器；
- 不把项目进展、实验事实、推测和外部结论混在一起；
- 不用目录树表达所有关系；
- 通过 Wiki Link 连接跨领域对象；
- 重要结论保留来源；
- 修改正式内容前备份；
- 默认不删除、不批量移动、不修改发布状态；
- 修改后运行检查和 Quartz 构建；
- 处理外部材料时优先使用 `$personal-wiki-curator`，先分析后应用；
- 每次材料任务结束后做 Skill 缺口复盘，只有可复用问题才固化；
- 未经明确要求不得 push；
- 不修改 PaperNotes、Notes 和 Blog。
